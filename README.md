# k8s-fastapi-starter

![CI](https://github.com/D-Chenkov/k8s-fastapi-starter/actions/workflows/ci.yml/badge.svg)

An end-to-end ML serving pipeline: a scikit-learn model is trained, tracked and versioned in **MLflow**, served behind a **FastAPI** API, containerized with **Docker**, and deployed to **Kubernetes** (minikube), with a **GitHub Actions CI** pipeline gating every push. It is small on purpose, but production-shaped: config and secrets are externalized, the container runs non-root, health probes and resource limits are set, and the served model is pulled from the MLflow Model Registry by alias so a new model can be promoted with no code change.

This is Portfolio Project #1 of my QA-Automation to ML-Engineer transition.

## Architecture

```
train_model.py ──> MLflow (track + register)          promote by moving the @champion alias
                        │                                         │
                        ▼                                         ▼
                 Model Registry  ◀── FastAPI /predict loads models:/iris-clf@champion
                                              │
                                              ▼
                                   Docker image (non-root)
                                              │
                                              ▼
                          Kubernetes: Deployment ─> ReplicaSet ─> Pods
                                    (Service, ConfigMap, Secret, probes, limits)

   GitHub Actions CI on every push:  test (pytest gate) ─> build image
```

## What it demonstrates

- Containerizing a FastAPI app with a cache-aware, non-root Dockerfile.
- The Kubernetes object chain (Deployment -> ReplicaSet -> Pod), Services via label selectors, scaling, and zero-downtime rolling updates + rollback.
- Externalized configuration (ConfigMap) and secrets (Secret), injected as env vars.
- Production hygiene: non-root container, readiness/liveness probes, resource requests/limits.
- MLflow experiment tracking + Model Registry, with alias-based promotion (`@champion`) so serving code never changes to ship a new model.
- A hermetic CI pipeline (GitHub Actions) that runs tests and builds the image on every push.

## Prerequisites

- Docker
- minikube (Docker driver) + kubectl
- Python 3.12 (for local dev / training; note the ML stack lags newer Python, so avoid 3.13+)

## Quickstart (Kubernetes / minikube)

```bash
# 1. Train the model artifact (writes app/model.joblib)
python train_model.py

# 2. Build the image and load it into minikube
docker build -t fastapi-app:v6 .
minikube image load fastapi-app:v6      # minikube's Docker driver can't see host images otherwise

# 3. Deploy everything (Deployment, Service, ConfigMap, Secret)
kubectl apply -f k8s/

# 4. Inspect the object chain
kubectl get deployment,replicaset,pods

# 5. Reach it (ClusterIP is internal, so port-forward)
kubectl port-forward service/fastapi-service 8080:80
```

Then, in another terminal:

```bash
curl localhost:8080/health
# {"status":"ok"}

curl -X POST localhost:8080/predict \
  -H "Content-Type: application/json" \
  -d '{"sepal_length":5.1,"sepal_width":3.5,"petal_length":1.4,"petal_width":0.2}'
# {"class_id":0,"class_name":"setosa"}
```

## Run locally without Kubernetes

```bash
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
python train_model.py
uvicorn app.main:app --port 8080     # serves the local app/model.joblib by default
```

Or with Docker Compose (also starts a Redis service to demonstrate Compose's service-name networking):

```bash
docker compose up --build
docker compose exec api getent hosts redis   # proves service-name DNS works
```

## Serving from the MLflow Model Registry

By default the app loads the local `app/model.joblib` (which keeps CI hermetic). Set `MODEL_URI` to load from the MLflow Model Registry instead:

```bash
# start a tracking server (holds the registry)
mlflow server --host 127.0.0.1 --port 5000
export MLFLOW_TRACKING_URI=http://127.0.0.1:5000

# register a run's model as "iris-clf" and alias the version as @champion (via UI or client)
# then point the app at it:
export MODEL_URI="models:/iris-clf@champion"
uvicorn app.main:app --port 8080
```

**Promoting a new model needs zero code change:** register the new model as a new version, move the `@champion` alias to it, and restart the app. `models:/iris-clf@champion` now resolves to the new version. The alias is a stable pointer (like a DNS name); the code never references a specific version.

## Configuration

| Variable | Purpose | Default |
|----------|---------|---------|
| `MODEL_URI` | Where to load the model: a local path (`app/model.joblib`) or a registry URI (`models:/iris-clf@champion`) | `app/model.joblib` |
| `MLFLOW_TRACKING_URI` | Address of the MLflow tracking/registry server (only needed for registry mode) | (unset) |
| `APP_VERSION` | Version string returned by `GET /` | `v6` |

`APP_GREETING` is injected from the ConfigMap and `API_KEY` from the Secret. The committed `API_KEY` is a **dummy dev value** (`dev-only-not-real`); real secrets are never committed (they'd go in an external secrets manager / encrypted-at-rest etcd + RBAC).

## Kubernetes details

- **Deployment -> ReplicaSet -> Pod:** you manage the Deployment; it creates a ReplicaSet per version; the ReplicaSet keeps the desired pod count alive (self-healing).
- **Service:** a `ClusterIP` that routes to pods by **label selector** (`app: fastapi-app`), load-balancing across them.
- **Probes:** `readinessProbe` on `/health` (no traffic until ready), `livenessProbe` on `/health` (restart if it stops responding).
- **Resources:** `requests` (guaranteed for scheduling) and `limits` (cap so one pod can't starve the node).
- **Security:** runs as a non-root user (`runAsNonRoot: true`, UID 1000).

Scale and roll:

```bash
kubectl scale deployment fastapi-app --replicas=3     # 3 pods behind one Service

# rolling update: build a new tagged image, load it, bump the tag in k8s/deployment.yaml, re-apply
kubectl apply -f k8s/
kubectl rollout status deployment/fastapi-app         # new ReplicaSet scales up as old scales down
kubectl rollout undo deployment/fastapi-app           # roll back (old ReplicaSet retained at 0)
```

Prefer editing `k8s/deployment.yaml` and re-applying over imperative `kubectl set image`, so the repo stays the single source of truth.

## Tests and CI

`tests/test_app.py` uses FastAPI's `TestClient` to check `/health` and `/predict`. The GitHub Actions workflow (`.github/workflows/ci.yml`) runs on every push and pull request:

1. checkout -> set up Python 3.12 -> install dependencies
2. `python train_model.py` (generate the model locally, so CI is **hermetic** and needs no MLflow server)
3. `pytest` (the test gate)
4. `docker build` (prove the image still builds)
